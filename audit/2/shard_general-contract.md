## DISCREPANCIES

### [GC-01] `spec/sse-query-invalidate-contract.md` client validation pipeline steps 3–5 don't match current behavior for target-discriminated signals

- **Area:** `spec/sse-query-invalidate-contract.md` — Client side → "Payload validation"
- **Type:** outdated-doc
- **Evidence:**
  - Spec validation pipeline step 3: *"Each object must have a `key` property that is an `Array`"*
  - Step 4: *"If `exact` is present, it must be `boolean`"*
  - Step 5: *"If `action` is present, it must be one of `'invalidate' | 'refetch' | 'remove'`"*
  - But `TanStackQuerySignal` uses `queryKey` (not `key`), has no `exact` property (uses TanStack-specific filters), and supports `'reset'` and `'cancel'` actions. `RTKQuerySignal` has no `key` property at all.
  - The structural validation rules in steps 3–5 were written for `GenericInvalidateSignal` only and would incorrectly reject valid `TanStackQuerySignal` or `RTKQuerySignal` payloads if enforced as described.
  - Actual client validation code was not read in this audit (not in scope). It's possible the implementation correctly handles target-discriminated signals while the spec text is outdated.
- **Discrepancy:** The spec text for client validation describes generic signals only but the wire protocol supports four signal types, some of which don't have a `key` property at all.
- **Which source is correct:** Implementation is likely more correct (it would be broken otherwise). Spec text needs updating to describe per-target-type validation rules.
- **Recommended fix:** Update the validation pipeline in `sse-query-invalidate-contract.md` to say: "For generic/SWR signals: must have `key` as an Array. For TanStack signals: must have `queryKey` as an Array. For RTK signals: must have `tags` as an Array. For all types: if present, `action` must be a value valid for that target's action union." Or: simplify to "structurally valid for the detected target type" and link to the type definitions.
- **Severity:** medium
- **Confidence:** medium (requires reading client validation.ts to confirm, which was not done)

### [GC-02] `SSEChannelOptions.connectionId` field is documented with conflicting JSDoc and spec description

- **Area:** `src/server/core/channel.ts` — `SSEChannelOptions.connectionId` JSDoc, `spec/sse-query-invalidate-contract.md`
- **Type:** contradiction
- **Evidence:**
  ```ts
  /**
   * The unique connection ID generated by the client and sent as `__restale_cid__`.
   * Transport adapters (`attachSSE`, `toSSEResponse`) extract this from the request URL
   * and pass it here automatically. You rarely need to set this manually.
   */
  connectionId?: string
  ```
  Contract spec on `attachSSE`: *"Extracts the `__restale_cid__` query parameter from the request URL and assigns it to the channel's `connectionId` property. Throws synchronously if the parameter is missing or empty."*
  
  The JSDoc says "You rarely need to set this manually" — but does not say what happens if it IS set manually (e.g. does it override the transport-extracted value?). The contract spec says transports "extract" it and pass it automatically. The JSDoc's "rarely" implies it's sometimes appropriate.
- **Discrepancy:** Minor inconsistency. The JSDoc implies manual setting is possible; the spec doesn't describe this use case. The implementation uses `options.connectionId ?? ''`, so a manually set value would take precedence over the transport's extraction if `createSSEChannel` is called directly (bypassing the transport).
- **Recommended fix:** Clarify JSDoc: "Normally set by transport adapters. You may set this in tests or when calling `createSSEChannel` directly without a transport adapter."
- **Severity:** low
- **Confidence:** high

### [GC-05] `docs/server.md` describes `broadcastByKey` key matching with "wraps scalar/array meta into a single-element array" — but the note in spec is slightly different

- **Area:** `docs/server.md` — `broadcastByKey` section, `spec/sse-query-invalidate-contract.md`
- **Type:** contradiction (minor)
- **Evidence:**
  - `docs/server.md` broadcastByKey: *"A channel receives the signal when its registered metadata matches or extends the signal key (when `exact: true`, keys must match exactly)"*
  - `spec/sse-query-invalidate-contract.md` `broadcastByKey`: *"If a channel's registered metadata is a scalar or plain object, it is auto-wrapped into a single-element array `[meta]` during key matching"*
  - `channel-group.ts` `broadcastByKey`: 
    ```ts
    const metaKey = Array.isArray(meta) ? meta : [meta]
    return matchesInvalidateSignalKey(metaKey, signal)
    ```
  - Docs says "matches or extends" — the actual matching semantics are prefix/exact per `matchesInvalidateSignalKey`. The docs description is a loose paraphrase, not technically wrong but less precise.
- **Discrepancy:** Minor precision gap in docs vs spec. Both describe the same behavior conceptually. Low risk.
- **Recommended fix:** Docs don't need changing. This is noted as confirmed-agreement with a slight wording difference.
- **Severity:** low
- **Confidence:** high

### [GC-06] `spec/restale-kit-connection-revocation-spec.md` doesn't mention Frame Guard's `onDeadline: 'revoke'` as a third revocation path

- **Area:** `spec/restale-kit-connection-revocation-spec.md`
- **Type:** spec-not-implemented (cross-spec gap)
- **Evidence:**
  - Revocation spec §1: describes `revokeWhere` and `revokeByConnectionId` as the two revocation mechanisms.
  - Frame Guard spec §6: *"`beforeFrame` returning `Close` is functionally equivalent to the integrator calling revocation directly"* and `onDeadline: 'revoke'` uses the same path.
  - Revocation spec does not mention Frame Guard at all.
- **Discrepancy:** The revocation spec is incomplete — it describes two of the three paths that lead to a terminal `revoke` frame. Frame Guard adds a third (and a fourth if you count `beforeFrame.close` separately from `onDeadline: 'revoke'`).
- **Recommended fix:** Add a §6 or appendix to the revocation spec: "Additional revocation paths via Frame Guard: `beforeFrame` returning `{ action: 'close' }` and `lifetime: { onDeadline: 'revoke' }` both route through the same `channel.revoke()` mechanism. See Frame Guard spec for details."
- **Severity:** low
- **Confidence:** high

### [GC-07] `FRAME_GUARD_DEFAULTS.RENEW_JITTER_FACTOR` is defined but has no server-side usage — it's a client-side constant misplaced in a shared module

- **Area:** `src/utils/constants.ts`, `src/client/core/sse-client.ts`
- **Type:** undocumented-behavior (architectural concern)
- **Evidence:**
  ```ts
  // constants.ts
  RENEW_JITTER_FACTOR: 0.2,
  ```
  - This is used only in `sse-client.ts` (`FRAME_GUARD_DEFAULTS.RENEW_JITTER_FACTOR`), which is the client package.
  - `FRAME_GUARD_DEFAULTS` as a whole is in `utils/constants.ts`, which is a shared utility. Other constants in `FRAME_GUARD_DEFAULTS` (`RENEW_MAX_ATTEMPTS`, `RENEW_RETRY_DELAY_MS`, `DEADLINE_MIN_FIRE_DELAY_MS`, `DEADLINE_JITTER_WINDOW_MS`) are server-side values.
  - Having a client-side constant in a server-side default set is conceptually odd. The spec §4.1.5 attributes this jitter to the client ("The client applies a small jitter window"), and it's used only in `sse-client.ts`.
- **Discrepancy:** Minor placement concern, not a bug. All constants sharing one module avoids duplication, but `RENEW_JITTER_FACTOR` is semantically a client-only constant.
- **Recommended fix:** Add a JSDoc comment to `RENEW_JITTER_FACTOR` clarifying it's client-side: "Client-side jitter fraction applied during renew confirmatory attempt spacing (spec §4.1.5). Server-side constants are above." Low priority.
- **Severity:** low
- **Confidence:** high
